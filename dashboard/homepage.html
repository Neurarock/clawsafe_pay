◆ Core Capabilities
Built for Security. Designed for Agents.
Every design decision prioritises keeping private keys offline and giving humans the final say on every transaction.
🔒

True Self-Custody

Private keys never leave your environment. No custodial risk. No third-party key management. You own your keys, always.

❄️

Cold Storage Architecture

The signer service runs air-gapped from the internet-facing API. Keys are loaded once and never transmitted over the wire.

✅

Human-in-the-Loop 2FA

Every transaction requires explicit Telegram approval. AI agents propose; humans authorise. No autonomous signing without consent.

⛓️

Multi-Chain Registry

Pluggable chain architecture supporting EVM chains, Solana, Bitcoin, Zcash, and Cardano. Add new chains via a simple registry interface.

🛡️

Policy Engine

Configurable spend limits, recipient allowlists, gas policy checks, and injection detection. Defence in depth at every layer.

🤖

Agent-Native API

REST API designed for LLM tool-use. Submit a payment intent in one call. The pipeline handles building, reviewing, signing, and broadcasting.

◆ Payment Flow
From Intent to On-Chain in Four Steps
The pipeline transforms a simple payment intent into a fully signed, broadcast, and confirmed on-chain transaction.
1

Agent Submits Intent

The AI agent sends a payment intent — recipient, amount, chain, and asset — to the publisher API.

2

Build & Review

The publisher builds a draft transaction, applies policy checks, and optionally runs an LLM reviewer for risk analysis.

3

Human Approval

The signer service sends a Telegram message with transaction details. The human taps Approve or Reject.

4

Sign & Broadcast

Upon approval, the transaction is signed locally with the cold-stored key and broadcast to the target chain.

◆ System Design
Microservice Architecture
Four isolated services with clear security boundaries. Each service has its own database and authentication layer.
┌─────────────────────────────────────────┐ │ AI Agent (OpenClaw / LLM) │ │ │ POST /intent │ │ ▼ │ │ Publisher Service :8002 │ │ ├── Build DraftTx │ │ ├── Policy Engine │ │ ├── Reviewer Service :8003 │ │ │ └── LLM risk analysis │ │ ▼ │ │ Signer Service :8001 │ │ ├── Chain Registry lookup │ │ ├── User Auth :8000 (Telegram) │ │ ├── 🔐 Local signing (cold key) │ │ └── Broadcast to chain │ └─────────────────────────────────────────┘
🔑

Key Isolation

Private keys exist only within the signer service process. The publisher and reviewer services never see them.

🔗

HMAC Authentication

Inter-service calls are authenticated with HMAC-SHA256 signatures. No service trusts another implicitly.

📊

Full Audit Trail

Every state transition is persisted to SQLite with timestamps. Complete traceability from intent to on-chain confirmation.

⚡

Chain Agnostic

The chains registry abstracts provider, builder, and signer per chain family. Adding a new chain is one Python module.
